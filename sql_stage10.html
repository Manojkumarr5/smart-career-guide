Step 10: Understand Keys and Constraints

In this stage you learn how constraints like PRIMARY KEY, FOREIGN KEY, NOT NULL, UNIQUE, CHECK, and DEFAULT help keep your data clean and consistent.

You also connect these ideas to basic normalization, where a good table design avoids unnecessary repetition and strange update problems.


1Ô∏è‚É£ Keys and constraints ‚Äì quick idea

A constraint is a rule the database enforces on a column or table, such as ‚Äúthis column cannot be empty‚Äù or ‚Äúthese values must be unique‚Äù.

  • Keys like PRIMARY KEY and FOREIGN KEY are special constraints that manage identity and relationships between tables.
  • Other constraints such as NOT NULL, UNIQUE, CHECK, and DEFAULT control allowed values and reduce bad data.

2Ô∏è‚É£ PRIMARY KEY and FOREIGN KEY

PRIMARY KEY uniquely identifies each row in a table and cannot be NULL; a table has exactly one primary key, which may be single‚Äëcolumn or composite. 

-- students with primary key
CREATE TABLE students (
    id      INT PRIMARY KEY,
    name    VARCHAR(100) NOT NULL,
    branch  VARCHAR(50),
    cgpa    DECIMAL(3,1)
);

-- marks table with foreign key that references students.id
CREATE TABLE marks (
    id         INT PRIMARY KEY,
    student_id INT,
    subject    VARCHAR(50),
    score      INT,
    FOREIGN KEY (student_id) REFERENCES students(id)
);

FOREIGN KEY ensures that every student_id in marks must exist in students.id, which protects the relationship between the tables.

3Ô∏è‚É£ NOT NULL, UNIQUE, CHECK, DEFAULT

These constraints control what values can be stored in a column and help avoid common data entry mistakes. 

CREATE TABLE users (
    user_id    INT PRIMARY KEY,
    email      VARCHAR(100) UNIQUE NOT NULL,
    full_name  VARCHAR(100) NOT NULL,
    age        INT CHECK (age >= 18),
    country    VARCHAR(50) DEFAULT 'India'
);
  • NOT NULL: value cannot be empty, so a row must supply something for that column.
  • UNIQUE: all values in the column must be different, often used for emails or usernames.
  • CHECK: value must satisfy a condition, such as age greater than or equal to 18.
  • DEFAULT: auto‚Äëfills a value if none is provided, such as default country.

These rules are enforced by the database itself so even if application code has bugs, invalid data is blocked.

4Ô∏è‚É£ Inline vs table-level constraint syntax

Constraints can be written inline with column definitions or at the end of the CREATE TABLE statement as separate table‚Äëlevel clauses. 

-- inline style
CREATE TABLE example1 (
    id    INT PRIMARY KEY,
    email VARCHAR(100) UNIQUE
);

-- table-level style
CREATE TABLE example2 (
    id    INT,
    email VARCHAR(100),
    CONSTRAINT pk_example2 PRIMARY KEY (id),
    CONSTRAINT uq_example2_email UNIQUE (email)
);

Both styles work; naming constraints with CONSTRAINT is helpful for debugging and for understanding error messages in larger projects.

5Ô∏è‚É£ Normalization ‚Äì basic link to constraints

Normalization such as 1NF, 2NF, and 3NF is a design approach that reduces redundancy and avoids strange update or deletion anomalies.

  • 1NF: each column holds atomic values and tables avoid repeating groups of columns.
  • 2NF/3NF: every non‚Äëkey column depends on the whole key and not on other non‚Äëkey columns, which removes partial and transitive dependencies.

Good keys and constraints support normalization by clearly defining primary keys and preventing duplicated or inconsistent data.
