Step 16: Use Real Datasets for Analysis

In this stage you move from small toy tables to real datasets from the internet (for example, Kaggle) and practice writing analysis queries on them.

This builds the habit of understanding a schema, asking questions, and writing SQL that discovers insights from real-world data.


1Ô∏è‚É£ Find and choose a real dataset

Many public datasets are available for free and can be used to practice SQL without touching any production systems.

  • Kaggle: datasets on Olympics, e-commerce, movies, healthcare, and more.
  • Government and open data portals with demographics, transport, and finance data.
  • Prefer CSV files that are not too huge, roughly 10k‚Äì100k rows, to keep practice smooth.

Pick a topic you like such as sports, movies, or sales so exploring the data feels engaging instead of boring.

2Ô∏è‚É£ Load the dataset into your database

After downloading CSV files, create tables that match their columns and import the data into your SQL database. 

-- example: table for an orders dataset
CREATE TABLE orders (
    order_id      INT PRIMARY KEY,
    order_date    DATE,
    customer_id   INT,
    country       VARCHAR(100),
    sales_amount  DECIMAL(10,2),
    category      VARCHAR(100)
);

-- then use your DB tool or a script to import the CSV:
-- MySQL example:
--   LOAD DATA INFILE 'orders.csv'
--   INTO TABLE orders
--   FIELDS TERMINATED BY ','
--   ENCLOSED BY '"'
--   LINES TERMINATED BY '\n'
--   IGNORE 1 LINES;

Most database GUIs provide an ‚ÄúImport CSV‚Äù wizard so you can load files without writing BULK INSERT or COPY commands initially.

3Ô∏è‚É£ Ask analysis-style questions

Once your data is in tables, think like an analyst and ask about patterns, totals, and trends that matter for the dataset. 

-- example questions on an orders dataset:

-- total sales by country
SELECT country,
       SUM(sales_amount) AS total_sales
FROM orders
GROUP BY country
ORDER BY total_sales DESC;

-- monthly sales trend
SELECT DATE_TRUNC('month', order_date) AS month,  -- function varies by DB
       SUM(sales_amount) AS total_sales
FROM orders
GROUP BY DATE_TRUNC('month', order_date)
ORDER BY month;

-- top 5 customers by total sales
SELECT customer_id,
       SUM(sales_amount) AS total_sales
FROM orders
GROUP BY customer_id
ORDER BY total_sales DESC
LIMIT 5;

These types of queries resemble typical BI dashboards that aggregate sales by geography, customer, and time.

4Ô∏è‚É£ Mix filters, joins, and derived columns

Real datasets often span multiple tables, so you will join them and apply filters and computed columns to answer deeper questions. 

-- suppose you also have a customers table
CREATE TABLE customers (
    customer_id INT PRIMARY KEY,
    name        VARCHAR(200),
    segment     VARCHAR(100),
    region      VARCHAR(100)
);

-- average order size by customer segment
SELECT c.segment,
       AVG(o.sales_amount) AS avg_order_value
FROM orders    AS o
JOIN customers AS c
  ON o.customer_id = c.customer_id
GROUP BY c.segment
ORDER BY avg_order_value DESC;

-- orders only for a specific region and category
SELECT o.order_id,
       o.order_date,
       c.region,
       o.category,
       o.sales_amount
FROM orders    AS o
JOIN customers AS c
  ON o.customer_id = c.customer_id
WHERE c.region  = 'Europe'
  AND o.category = 'Electronics';

This mirrors daily analyst work: joining fact tables with dimension tables and then aggregating and filtering the results.
