Step 5: Understand INSERT, UPDATE, DELETE

In this stage you learn the main DML (Data Manipulation Language) commands: INSERT, UPDATE, and DELETE, which let you add, change, and remove rows in your tables.

You will practice them on your students table so that CRUD (Create, Read, Update, Delete) becomes natural.


1Ô∏è‚É£ INSERT ‚Äì add new rows

The INSERT statement adds new records to a table by specifying the table name, the target columns, and matching values. 

-- Insert a single student
INSERT INTO students (id, name, branch, cgpa)
VALUES (4, 'Kiran', 'ME', 8.9);

-- Insert multiple students (if your DB supports this syntax)
INSERT INTO students (id, name, branch, cgpa)
VALUES
  (5, 'Salma', 'CSE', 9.2),
  (6, 'Arjun', 'EEE', 7.5);

After inserting, run SELECT * FROM students; to confirm that the new rows appear as expected.

2Ô∏è‚É£ INSERT without listing columns (and why to avoid it)

You can also insert by providing values in the table‚Äôs column order without listing column names, but this is risky if the table structure changes. 

-- Only works if you follow the exact column order in the table
INSERT INTO students
VALUES (7, 'Deepa', 'CSE', 8.0);

Best practice is to always list columns in INSERT, especially in real projects, so your SQL is less fragile.

3Ô∏è‚É£ UPDATE ‚Äì modify existing rows

UPDATE changes values of existing rows; a WHERE clause usually targets specific rows, and if omitted it updates all rows. 

-- Increase Kiran's cgpa
UPDATE students
SET cgpa = 9.1
WHERE id = 4;

-- Change branch for a specific student
UPDATE students
SET branch = 'CSE'
WHERE name = 'Arjun';

Always double‚Äëcheck the WHERE condition before running UPDATE; a missing WHERE will modify every row in the table.

4Ô∏è‚É£ DELETE ‚Äì remove rows

DELETE removes rows from a table; with WHERE you delete chosen records, and without WHERE you delete all rows. 

-- Delete a single student by id
DELETE FROM students
WHERE id = 6;

-- Delete all students in a particular branch
DELETE FROM students
WHERE branch = 'EEE';

Avoid running DELETE FROM table_name; on important data, because it clears the entire table; use it only with care in test databases.

5Ô∏è‚É£ CRUD flow: INSERT ‚Üí SELECT ‚Üí UPDATE ‚Üí SELECT ‚Üí DELETE ‚Üí SELECT

A good way to build confidence is to perform a full CRUD cycle on a practice row and watch how the table changes after each step. 

-- 1) CREATE (insert a new row)
INSERT INTO students (id, name, branch, cgpa)
VALUES (10, 'TestUser', 'TEST', 5.0);

-- 2) READ (check it)
SELECT * FROM students WHERE id = 10;

-- 3) UPDATE (fix data)
UPDATE students
SET cgpa = 7.0, branch = 'CSE'
WHERE id = 10;

-- 4) READ (check again)
SELECT * FROM students WHERE id = 10;

-- 5) DELETE (remove practice row)
DELETE FROM students
WHERE id = 10;

In many systems, DML changes are finalized with COMMIT and can be undone with ROLLBACK before committing, which you will see in the transactions stage.
